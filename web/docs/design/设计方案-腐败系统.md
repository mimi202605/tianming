# 设计方案 · 腐败系统（吏治浊清）

> 第三官方变量。与帑廪、内帑并列，构成"治体三维"。  
> **定位**：不是独立的游戏目标，而是所有子系统（财政/户口/军事/工程/司法）的"漏损层"与"信号失真层"。  
> **隐喻**：不是一个分数，而是一层薄膜——它让朝堂所见与天下实情产生裂痕。

---

## 零、核心哲学

### 0.1 三条铁律

1. **腐败不是惩罚，是物理现象**  
   不对玩家"扣分"，而是让腐败自然传导为实征率、军饷到位率、工程合格率等具体数字。玩家看到的是"漏损"，不是"腐败-5"。

2. **腐败的真实度 ≠ 玩家看到的腐败**  
   真实腐败（trueIndex）与玩家感知腐败（perceivedIndex）分离。监察力度决定二者差距。肃贪的第一个敌人是"不知道"。

3. **肃贪必有代价**  
   无副作用的肃贪是作弊。酷吏→冤狱；党争揭发→党祸；特务机构→反噬；俸禄改革→财政压力。没有"免费的清廉"。

### 0.2 与"制度设计"的关系

腐败系统与制度设计器深度绑定：  
- 玩家可用诏书创设**监察机构**（御史台、都察院、提刑按察使、巡按御史、锦衣卫…）  
- 每个监察机构自带**侦测半径**、**独立性**、**腐败率**三个参数  
- 监察机构本身也会腐败（"刑部尚书贪污案""御史收贿案"）——"谁来监督监督者"

### 1.2 真实值 vs 感知值

```javascript
// perceived 由 true + 监察力度 + 随机偏差 决定
function updatePerceived(dept) {
  var sup = getSupervisionFor(dept);        // 对该部门的监察力度 0-100
  var trueV = dept.true;
  var maxDelta = (100 - sup) * 0.4;         // 监察越弱，偏差越大
  var delta = (Math.random() - 0.3) * maxDelta;  // 负向偏差更可能（报喜不报忧）
  dept.perceived = clamp(trueV + delta, 0, 100);
}
```

**关键设计**：默认偏差 **负向**（perceived < true）。玩家接收到的永远是"天下承平""吏治尚可"的报告——直到某个事件撕开裂痕。

## 二、来源机制（9 类）

### 2.1 俸禄过低（lowSalary）

**历史依据**：  
- 明代官员正俸极薄，胥吏无俸，形成"三年清知府，十万雪花银"的结构性贪腐  
- 清代雍正设"养廉银"以应对

**公式**：
```javascript
function calcLowSalaryCorruption() {
  var avgSalary = getAvgOfficialSalary();       // 人均俸禄（文）
  var livingCost = getOfficialLivingCost();     // 官员家庭生活成本
  var ratio = avgSalary / livingCost;
  
  // ratio=1.0 → 0 腐败
  // ratio=0.5 → +15
  // ratio=0.3 → +25
  var base = ratio >= 1.0 ? 0 : (1.0 - ratio) * 30;
  
  // 胥吏无俸放大器
  if (GM.officialSystem.clerksPaid === false) base *= 1.5;
  
  return base;
}
```

**反作用**：大幅提薪（养廉银）→ 财政支出↑ → 帑廪压力↑

### 2.2 监察松弛（laxSupervision）

**公式**：
```javascript
function calcLaxSupervision() {
  var sup = GM.corruption.supervision.level;
  // sup=0 → +20
  // sup=50 → +5
  // sup=100 → 0
  return Math.max(0, (60 - sup) * 0.4);
}
```

**触发事件**：监察机构缺员率 > 30% 时，触发"御史缺额，弹章日稀"警示。

### 2.3 战时/灾时加派（emergencyLevy）

**历史依据**：  
- 明末辽饷、剿饷、练饷三饷加派 → 地方官借机"浮收"  
- 灾年临时征派 → 截留赈济

**触发条件**：`activeWar || activeFamine || activePlague || specialLevy`

```javascript
function calcEmergencyLevy(context) {
  var boost = 0;
  if (context.activeWar)    boost += 5;
  if (context.activeFamine) boost += 4;
  if (context.activePlague) boost += 3;
  if (context.specialLevy)  boost += context.specialLevy.rate * 10;
  return boost;
}
```

**衰减**：事件结束后每回合 -1，但"浮收习惯"留一条尾巴（永久+2，需专项整顿）。

### 2.4 卖官鬻爵（officeSelling）

**历史依据**：东汉灵帝西园卖官、清末捐纳

**玩家触发**：  
- 财政紧急时，玩家可诏令"开捐纳"  
- AI 辅政大臣在帑廪枯竭时会奏请

**公式**：
```javascript
function calcOfficeSelling() {
  if (!GM.juanna || !GM.juanna.active) return 0;
  var scale = GM.juanna.monthlyIncome / GM.guoku.monthlyIncome;
  // 捐纳收入占比 5% → +5；30% → +25

}
```

**传导**：卖官 → 官员素质↓ → 贤能↓ → AI 决策质量↓

```javascript
function calcNepotism() {
```

### 2.6 宠臣/宦官/外戚（innerCircle）

**历史依据**：和珅、刘瑾、魏忠贤、梁冀、杨国忠

**触发**：当某角色 `influence > 80` 且 `integrity < 30` 且 `empPrestige < targetChar.loyalty` 时激活

```javascript
function calcInnerCircleCorruption() {
  var active = characters.filter(c => 
    c.influence > 80 && c.integrity < 30 && isImperialFavorite(c)
  );
  return active.reduce((sum, c) => sum + (c.influence - 70) * 0.3, 0);
}
```

**特殊机制**：宠臣腐败有**阴影扩散**——受其庇护的下属腐败容忍度提升 50%。

### 2.7 冗官冗员（redundancy）

**历史依据**：北宋冗官（真宗朝万人，徽宗朝4.3万人）

```javascript
function calcRedundancy() {
  var actualOfficials = countAllOfficials();
  var idealCount = calcIdealOfficialCount(GM.population, GM.territorySize);
  var excess = actualOfficials / idealCount;
  // excess=1.0 → 0
  // excess=1.5 → +8
  // excess=3.0 → +20
  return Math.max(0, (excess - 1.0) * 12);
}
```

### 2.8 制度漏洞（institutional）

由制度设计器检测：玩家新设机构若缺少"考核""监察""任期"条款，此项累加。

```javascript
function calcInstitutionalGap() {
  var gap = 0;
  for (var inst of GM.dynamicInstitutions) {
    if (!inst.hasAudit)         gap += 1;
    if (!inst.hasTermLimit)     gap += 1;
    if (!inst.hasSupervision)   gap += 2;
    if (inst.budget > 100000 && !inst.hasAccountability) gap += 2;
  }
  return Math.min(gap, 15);
}
```

### 2.9 诏书巨额一次性支出（lumpSumSpending）

**设计动机**：历史上最肥的贪腐温床不是日常行政，而是**"过手皆湿"**的大项目。巨额、紧急、层级多、监察不完善的一次性开支，等于是把整条官僚链泡在蜜罐里。

**历史依据**：
- **隋炀帝**：营建东都、开凿运河、三征高句丽——每项均超年入，工料被层层盘剥，民怨沸腾
- **唐玄宗**：兴建华清宫、安史之乱军费——节度使借机自肥
- **宋真宗**：封禅泰山（约年入 20%）、大建玉清昭应宫——王钦若、丁谓借机敛财
- **宋徽宗**：花石纲（苏州应奉局年耗巨万）——朱勔一门皆贵
- **明代**：三大殿火灾后重建、郑和下西洋、修筑长城——工部、户部侵蚀严重
- **清代**：康熙三征噶尔丹、乾隆"十全武功"、六下江南、河工——历任河道总督人人巨富（"河工银"定律）
- **和珅案**：抄没家产约等于清廷 15 年国库收入，巨部分来自经办巨工、军需的盘剥

**核心公式**：
```javascript
function calcLumpSumCorruption(expenditure, context) {
  var annualIncome = GM.guoku.annualIncome || 1;
  var ratio = expenditure.amount / annualIncome;

  // 5% 以下不触发
  if (ratio < 0.05) return { corruption: 0, depts: {} };

  // 非线性基础曲线：
  //   5% → 2；10% → 5；20% → 12；30% → 22；50% → 40；100% → 80
  var base = 10 * Math.pow(ratio - 0.04, 1.3);

  // 层级放大：中央→省→府→县→工头，每多一层 +15%
  var layers = expenditure.executionLayers || 3;
  if (layers > 3) base *= (1 + (layers - 3) * 0.15);

  // 紧急性放大（时间紧 → 审核宽松）
  if (expenditure.urgent) base *= 1.25;

  // 类型系数
  var typeMult = {
    infrastructure: 1.2,   // 工程类最肥（物料可偷工、用工可虚报）
    military:       1.15,  // 军需侵吞
    reward:         0.8,   // 赏赐较难截留，但内廷可扣
    relief:         1.1,   // 赈济被截留是历代痼疾
    ritual:         1.0,   // 祭祀、庆典
    diplomacy:      0.9    // 外交币帛
  }[expenditure.type] || 1.0;
  base *= typeMult;

  // 监察减免
  var audit = GM.corruption.supervision.level;
  if (expenditure.hasDedicatedAudit)   audit += 25;  // 专项稽查
  if (expenditure.decreeHasOversight)  audit += 15;  // 诏书含监察条款
  if (expenditure.stagedApproval)      audit += 10;  // 分期拨付
  if (expenditure.publicTally)         audit += 10;  // 清册榜示
  var auditMult = Math.max(0.25, 1 - audit / 150);

  var totalCorruption = base * auditMult;

  // 部门分配（不同项目类型流向不同部门）
  var deptDistribution = {
    infrastructure: { central: 0.35, provincial: 0.50, fiscal: 0.15 },
    military:       { military: 0.50, central: 0.25, fiscal: 0.25 },
    reward:         { imperial: 0.50, central: 0.30, fiscal: 0.20 },
    relief:         { provincial: 0.55, fiscal: 0.30, central: 0.15 },
    ritual:         { imperial: 0.40, central: 0.40, fiscal: 0.20 },
    diplomacy:      { central: 0.50, imperial: 0.30, fiscal: 0.20 }
  }[expenditure.type] || { central: 0.4, provincial: 0.3, fiscal: 0.3 };

  var result = { corruption: totalCorruption, depts: {} };
  for (var d in deptDistribution) {
    result.depts[d] = totalCorruption * deptDistribution[d];
  }

  // 集团凝聚风险：巨项目易催生"盐政党""河工党""军需党"
  if (ratio > 0.25) {
    result.factionFormationRisk = Math.min(0.85, (ratio - 0.15) * 1.5);
  }

  // 民怨叠加（若项目与民生直接相关）
  if (expenditure.directPeopleBurden) {
    result.peopleResentmentSpike = ratio * 15;
  }

  return result;
}
```

|------|-----------|---------|------|
| < 5% | 0 | 寻常政务 | 不触发 |
| 5-10% | 2-5 | 中型水利、岁修 | 可控 |

**衰减与清算**：
```javascript
function decayLumpSumIncident(incident, context) {
  // 项目进行中：腐败缓慢爬升到峰值
  if (incident.status === 'active') {
    var monthsElapsed = context.currentTurn - incident.startTurn;
    if (monthsElapsed < incident.expectedDuration) {
      incident.currentCorruption = Math.min(
        incident.peakCorruption,
        incident.currentCorruption + incident.peakCorruption * 0.02
      );
    }
  }

  // 项目完工或被审计：进入自然衰减（每月 8%）
  if (incident.status === 'completed' || incident.status === 'audited') {
    incident.currentCorruption *= 0.92;
    if (incident.currentCorruption < 0.5) incident.status = 'closed';
  }

  // 爆出丑闻：进入事件处理，由选项决定衰减速度
}
```

**触发与接入**（与诏书实施阶段对接）：
```javascript
function onDecreeEnacted(decree, implementationPlan) {
  var totalCost = implementationPlan.lumpSumCost || 0;
  if (totalCost < GM.guoku.annualIncome * 0.05) return;  // 小额不触发

  var incident = {
    id: genId(),
    decreeId: decree.id,
    name: decree.title || implementationPlan.projectName,
    type: classifyExpenditureType(decree, implementationPlan),
    amount: totalCost,
    ratioToAnnual: totalCost / GM.guoku.annualIncome,
    executionLayers: implementationPlan.executionLayers || 3,
    urgent: decree.urgencyFlag || implementationPlan.urgent || false,
    // 由 AI 解析诏书文本自动识别监察条款
    hasDedicatedAudit:   implementationPlan.auditAssigned || false,
    decreeHasOversight:  /遣.*稽查|专差监|分期验收|清册|榜示/.test(decree.text),
    stagedApproval:      implementationPlan.stagedPayment || false,
    publicTally:         /榜示|公示|清册/.test(decree.text),
    startTurn: GM.turn,
    expectedDuration: implementationPlan.durationMonths || 12,
    directPeopleBurden: implementationPlan.corveeScale > 50000,
    status: 'active'
  };

  var result = calcLumpSumCorruption(incident, context);
  incident.peakCorruption = result.corruption;
  incident.currentCorruption = result.corruption * 0.3;  // 初始显现 30%
  incident.depts = result.depts;
```

## 七、与其他变量联动

### 7.1 腐败 → 帑廪
- 实征率：`1 - (fiscal + provincial)/2 × 0.007`
- 例：腐败 0 → 100% 入库；腐败 70 → 约 51% 入库

### 7.2 腐败 → 内帑
- 内廷腐败直接侵吞：`neitang.monthlyIncome × imperial/100 × 0.5`

### 7.3 腐败 → 民心
- 漏损传导：税收漏损 → 浮收加派 → 民心↓
- 冤狱传导：judicial.true × 0.3 每回合扣民心
- 直接感知：民众能观察"小吏气焰""富户奢靡"

### 7.4 腐败 → 威望
- 大型揭发案有双向作用：
  - 有效处置：威望+10~30（肃贪威名）
  - 掩盖/处理失当：威望-15~30（纵容腐败）

### 7.5 腐败 → 皇权
- 宠臣式腐败：`imperial.true > 60` 时，皇权被架空（决策权落入内廷）
- 强硬肃贪：皇权+10~20（但士人离心）

### 7.6 腐败 → 户口
- 豪强隐匿放大：`(provincial + fiscal)/2 > 50` 时，隐户每年+1%
- 胥吏勒索：`provincial > 60` 时，小民逃亡加速

### 7.7 腐败 → 军队
- 战力折算：`effectiveArmy = nominalArmy × (1 - military/100 × 0.5)`
- 战败概率上浮：`military × 0.005 每场战斗`

### 7.8 腐败 → 制度设计 / 诏书系统
- 制度漏洞检测：玩家新立机构若缺关键条款 → institutional +2~8
- 机构自身腐败：`每个机构.corruption` 独立演化，最终也需清理
- **诏书巨额支出**：
  - 二阶段奏疏时，若估计支出 > 5% 年入，户部自动预警并列出甲乙丙丁四案
  - AI 解析诏书中"遣XX稽查""分期""清册"等关键词，自动减免腐败
  - 多项同时进行（如隋炀帝式叠加）→ 累积判定"糜费天下"级事件
  - 项目完工后进入 60-120 月的衰减期，期间可能被事后审计揭发丑闻

### 7.9 腐败 → 角色
- 角色 `integrity` 低的，会因腐败环境加速堕落：
  - `environCorruption > 50` → 每年 integrity -2~5
  - 反之：清明环境中，贪官亦会收敛

### 7.10 腐败 → 贤能
- 卖官鬻爵 → 新入仕官员能力折扣
- 裙带荫补 → 平均能力↓
- 酷吏肃贪 → 人才寒心（integrity高者辞官潮）

### 9.7 风闻录事四类自动生成规则

每回合末，AI 推演后调用风闻生成器，产出 `告状 / 风议 / 密札 / 耳报` 四类条目写入 `GM.evtLog`。所有生成都通过 `addEB(type, text, {credibility, subject, source, ref})` API 落地。

#### 9.7.1 告状（登闻鼓/叩阍）

**触发公式**：
```javascript
var prob = (localCorruption - 30) / 100
         * GM.corruption.countermeasures.publicAppeal.accessibility
         * 2;
if (GM.activeFamine || GM.activeWar || GM.activePlague) prob *= 1.5;
if (Math.random() < Math.min(0.8, prob)) generateAccusation();
```

**目标选择**：
- 筛选 `regionCorruption > 40` 的地方
- 随机选一区，再随机选该区一名地方官（优先 `integrity < 50`）

**罪名模板**：加派火耗 / 冤狱苛刑 / 与豪强勾结 / 勒索差役 / 侵占田产 / 截留赈济 / 卖放罪犯

**Credibility 判定**：
| 证据 | 级别 |
|------|------|
| 账册/实物/多证人 | `high` |
| 部分凭据 | `medium` |
| 仅民怨无凭 | `low` |
| 十人以上联名 | 强制升 `high` |

**频率上限**：每回合 ≤ 2 条

#### 9.7.2 风议（士林舆论）

**触发**：事件驱动 + 角色热度驱动
- **事件驱动**：经筵/朝议/文会/春闱/秋闱/殿试/大典后必生成 1-3 条
- **热度驱动**：某角色 `fame ≥ 80 || infamy ≥ 60` 时每 3-5 回合 1 条
- **党争驱动**：`partyStrife > 60` 时频率 ×2

**目标选择**：
- 优先朝堂高层（丞相、尚书、侍郎级）
- 或正值风头上的地方官
- 近期有争议政策/工程的决策者

**议题类型**：
- `议政` — 对某政策的赞弹
- `议人` — 对某官员的风评
- `议时` — 对朝局的感喟

**Credibility 分布**：通常 `low` 或 `medium`，**极少** `biased`（士林中立性强）。多人同论 + 有凭据 → `medium`；孤例流言 → `low`。

**频率上限**：每回合 ≤ 3 条

#### 9.7.3 密札（门生故吏）

**触发前提**：
- 存在角色关系网（师生/同乡/同年/姻亲/荐主/门生）
- 关系链中某节点处于关键状态：升迁 / 弹劾 / 失势 / 病危 / 外放 / 赴任

**基础概率**：30% × 关系亲密度系数（0.5-1.5）

**内容类型与 credibility**：
| 类型 | 描述 | credibility |
|------|------|------------|
| `inside-info` | 透露内情（对主角有利） | `medium` |
| `plea` | 求情庇护 | `medium` |
| `betrayal` | 背刺揭发换前程 | `biased`（自利） |
| `warning` | 警告风险 | `medium` |
| `parting` | 告别（即将离任/病殁） | `high` |
| `solicit` | 图谋结纳 | `low`（多试探） |

**频率上限**：每回合 ≤ 2 条

#### 9.7.4 耳报（厂卫/宦官）

**前提**：玩家已设立特务机构（锦衣卫 / 东厂 / 西厂 / 提督太监 / 内行厂）

**生成量**：
```javascript
var count = Math.floor(inst.radius / 30) + 1;
count = Math.min(5, count);
```

**目标选择偏见**：
- 机构 `independence < 30` → 倾向报"皇帝喜欢听的"（打击对手派系）
- 机构 `independence > 60` → 敢报宗室/宠臣/外戚
- 机构自身腐败 > 60 → 开始保护受贿对象，诬告无辜

**内容类型**：
- `lifestyle` — 奢侈生活（姬妾/园亭/宴席）
- `gathering` — 私下结集
- `correspondence` — 通私书
- `business` — 涉商
- `disloyalty` — 不敬皇权之语（最严重）
- `suspicious-travel` — 非公出行
- `occult` — 求神问卜/巫蛊（**谋逆级罪名**）

**Credibility**：**恒为 `biased`**。原因：厂卫立功心切、善于夸大，对无辜诬告风险高。

**反噬机制**：若机构腐败 > 60 持续 > 12 月 → 触发"厂卫反咬"事件（被揭发受贿/构陷）。

**频率上限**：每回合 ≤ 5 条（受机构数量约束）

#### 9.7.5 噪声控制与优先级

**每回合风闻总量上限：10 条**（全部类型合计）。

若生成数 > 10：按下列优先级截断末尾：

1. `耳报`（固定周期，不可略）
2. `风议`（事件后必发）
3. `告状`（概率触发）
4. `密札`（偶发）

多个条目涉及同一角色时，**合并为一条**（避免"同一天五人指向张某"）。

#### 9.7.6 点击交互（风闻录事流水 → 弹窗详情）

点击风闻条目 → 弹出详情卡：

```
╔════════════════════════════════════╗
║  [告状] · 参考                      ║ ← credibility tag
║  青州民王二状告知府张宗道加派火耗   ║
║  证据：账册两本 · 证人三人          ║
║  民变24 | 党争67 | 威望55           ║
║ ─────────────────────────────────── ║
║  玩家处置：                         ║
║  [派钦差彻查] [付都察院勘]          ║
║  [召王二入对] [下户部核账]          ║
║  [留中不发]   [手诏问张某]          ║
║  [付六部会议]                       ║
╚════════════════════════════════════╝
```

**行动可用性规则**：
- `credibility = high | medium` → 全部行动可用
- `credibility = low` → 仅显示"留中不发 / 手诏问 / 派员细查"（不可直接处罚）
- `credibility = biased` → 标"需核验"，行动按钮灰显（点击提示"风闻偏颇，不可直诛"）

**防止凭风闻滥权**：任何导致角色罢黜/处死的行动，credibility 必须 ≥ `medium`，否则需先"付都察院勘"→ 数回合后升级证据。

| 模式 | 告状频率 | 风议频率 | 密札频率 | 耳报偏颇度 | 显示延迟 |
|------|---------|---------|---------|-----------|---------|
| 严格史实 | ×0.6 | ×0.7 | ×0.8 | 更偏颇 | 1-2 回合延迟 |
| 轻度史实 | ×1.0 | ×1.0 | ×1.0 | 正常 | 即时 |
| 演义模式 | ×1.3 | ×1.3 | ×1.2 | 偏颇度减轻 | 即时 |

#### 9.7.8 AI 生成文本接入

所有四类条目的文本由 AI（`callAI`）生成，system prompt 要求：
- **文言小笺风格**（不长，30-80 字为佳）
- 带具体姓名、地点、金额（提升真实感）
- 符合该类角色身份（宦官语气谄媚；士林语气矜持；门生语气亲切或背刺）

### 10.3 官员名册中的标记（按游戏模式分化）

#### 演义模式
角色卡片直接显示廉洁指示图标：
- 🏮 清廉（integrity > 80）
- 🏷 合格（integrity 50-80）
- 💰 嫌疑（integrity 30-50，有弹章在案）
- ⚠ 已揭发（integrity < 30，有案件）

（仍受监察覆盖约束：若该官员所在部门/区域无监察机构覆盖，图标显示"未察" ❓）

#### 严格史实模式 / 轻度史实模式
**不显示任何廉洁图标**。角色卡片上只有职位、能力、关系、派系等"可观察之事"。玩家必须通过下列渠道自行拼凑官员的清浊：

| 渠道 | 来源 | 可靠度 | 用时 |
|------|------|-------|------|
| 言官弹章 | 每月奏疏栏定期出现 | 中（可能党争陷害） | 被动接收 |
| 钦差调查 | 玩家主动派遣 | 高 | 3-6 月 |
| 登闻鼓 / 叩阍 | 民间告状，概率触发 | 低（多流言） | 被动 |
| 账目清查 | 派户部核对 | 高（需监察配合） | 1-2 月 |
| 士林风议 | 文会、朝议中的舆论线索 | 中低 | 被动 |
| 门生故吏泄密 | 角色关系链自然暴露 | 中 | 被动 |
| 私访 / 微服 | 特殊指令（代价：皇权声望） | 高 | 即时 |
| 宦官耳报 | 内廷上报（若启用东厂式机构） | 高但偏颇 | 被动 |

**严格史实模式额外限制**：  
- 即便查到"嫌疑"级证据，角色卡片也不自动打 💰 标记——玩家需自己在"史官密档"中手动记录（可用"私笔"功能备注）  
- 官员若未被正式立案弹劾，其腐败是"朝野尽知却无人肯言"的状态，符合历史上大量"**人人皆知严嵩贪，却无人敢奏**"的情境

#### 设计哲学
图标的"有无"是游戏模式的**认识论分水岭**：
- **演义模式**：玩家视角接近全知"天眼"——适合叙事沉浸、快速决策
- **史实模式**：玩家视角受"朝廷视野"限制——真实还原"皇帝是最后一个知道的人"的历史常态

这与 §0.1 第二条"真实腐败 ≠ 玩家看到的腐败"相呼应，使"监察投入"在史实模式成为真正有意义的战略资源。

## 十四、设计特色总结

1. **物理现象化**：腐败不是抽象分数，而是具象的"漏损层+信号失真层"——玩家看到的是实征率下降、军队空额、工程偷工，而不是"腐败+5"。

2. **真实值 vs 感知值**：肃贪的第一步是"知道"。监察力度越弱，玩家看到的数字越假。这让"整顿监察"本身成为一场博弈。

3. **所有反制皆有代价**：酷吏→冤狱，党争→党祸，特务→反噬，俸禄改革→财政压力。没有免费的清廉。

4. **与制度设计深度绑定**：玩家可创设任意监察机构，每个机构有独立性、覆盖、自身腐败三参数。监督者本身也会被监督。

5. **六部门独立演化**：中央/地方/军队/税司/司法/内廷各有不同腐败形态与源头，不是"一根棒子"。

6. **腐败集团凝聚**：持续高腐败 5 年 → 盘根错节集团 → 清剿成本指数上升。历史上的严党、阉党即此机制。

7. **AI 数据失真**：腐败高时，AI 看到的报表是被篡改的。玩家决策基于扭曲数据，这也是史上名君的真实处境。

8. **历史可变**：朝代开局预设了历史腐败水平，但玩家可以力挽狂澜或加速沉沦，由自己的政策决定。

9. **诏书 → 腐败闭环**：玩家每一道涉及巨额支出的诏书，都进入 lumpSumIncidents 追踪。项目规模、紧急度、层级数、诏书文本中的监察用语，共同决定后续几年的腐败累积曲线。二阶段奏疏的甲乙丙丁预警，让玩家在"速成"与"慎重"间做清晰权衡——而不是无声无息地自食恶果。
