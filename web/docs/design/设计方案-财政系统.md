## 决策 A 详细方案（已定稿）

### A.1 数据模型（C 决策后修订为三列）

存储位置：`GM.fiscal`

```js
GM.fiscal = {
  guoku:   { money: <Ledger>, grain: <Ledger>, cloth: <Ledger> },   // 帑廪·钱·粮·布
  neicang: { money: <Ledger>, grain: <Ledger>, cloth: <Ledger> },   // 内帑·钱·粮·布
  meta: {
    unitMoney: '贯',        // 从剧本读取
    unitGrain: '石',
    unitCloth: '匹',        // 新增
    lastSettleTurn: 0,
    historyDepth: 40,
    yearlyArchive: {}
  }
}
```

**布帛列显示规则**：stock=0 且无启用的 cloth 类税种时自动隐藏，保持顶栏清爽。

### A.2 单账（Ledger）结构

```js
{
  stock: 0,                // 当前库存，内部存"一单位"（1贯/1石），显示时 ÷10000 → 万
  lastTurnIn: 0,           // 上回合总入
  lastTurnOut: 0,          // 上回合总出
  thisTurnIn: 0,           // 本回合累计入（回合末→lastTurnIn 后清零）
  thisTurnOut: 0,          // 本回合累计出
  sources: {               // 本回合收入分解（tag→金额）
    '田赋': 0, '商税': 0, '盐铁': 0, ...
  },
  sinks: {                 // 本回合支出分解
    '俸禄': 0, '军饷': 0, '工程': 0, ...
  },
  history: [               // 近 depth 回合流水快照
    { turn, in, out, stockEnd, sources, sinks }
  ]
}
```

### A.3 单位与精度

- 内部全部按"一单位"存（`stock` 单位 = 1贯 或 1石）
- 显示层统一 `(stock / 10000).toFixed(1) + '万' + unit` → "帑廪 500.0万贯"
- 顶栏差值：`(thisTurnIn - thisTurnOut) / 10000` 加 ↑↓ 箭头

### A.4 history 深度动态计算

```js
var _dpv = _getDaysPerTurn();                         // 剧本每回合天数
var _depth = Math.max(12, Math.min(120, Math.ceil(730 / _dpv)));  // 约 2 年
GM.fiscal.meta.historyDepth = _depth;
```

- 1 天/回合 → 120（封顶）
- 15 天/回合 → 49
- 30 天/回合 → 25
- 90 天/回合 → 12（底线）

超出 depth 的老流水不丢，会在年度结算时归入 `meta.yearlyArchive[year]` 永久存档。

### A.5 初始化

`_ensureGMDefaults` 中：

```js
if (!GM.fiscal) {
  var fc = (sc && sc.fiscalConfig) || P.fiscalConfig || {};
  GM.fiscal = {
    guoku:   { money: _newLedger(fc.initial?.guokuMoney||0),  grain: _newLedger(fc.initial?.guokuGrain||0) },
    neicang: { money: _newLedger(fc.initial?.neicangMoney||0), grain: _newLedger(fc.initial?.neicangGrain||0) },
    meta: {
      unitMoney: fc.unit?.money || '贯',
      unitGrain: fc.unit?.grain || '石',
      lastSettleTurn: 0,
      historyDepth: /* A.4公式 */,
      yearlyArchive: {}
    }
  };
}
```

### A.6 统一读写 API（tm-utils.js 新增 Fiscal 对象）

```js
var Fiscal = {
  get: function(path)  { /* 'guoku.money' → stock */ },
  add: function(path, amount, tag) {
    var led = this._resolve(path);
    led.stock += amount;
    led.thisTurnIn += amount;
    led.sources[tag] = (led.sources[tag]||0) + amount;
  },
  spend: function(path, amount, tag) {
    var led = this._resolve(path);
    led.stock -= amount;   // 允许赤字（国库告罄由后续机制判断）
    led.thisTurnOut += amount;
    led.sinks[tag] = (led.sinks[tag]||0) + amount;
  },
  transfer: function(fromPath, toPath, amount, tag) {
    this.spend(fromPath, amount, '拨付'+tag);
    this.add(toPath, amount, '拨入'+tag);
  },
  settle: function(turn) {
    // 每回合末调用：thisTurn*→lastTurn*、存入 history、清零本回合累计
    _forEachLedger(function(led){
      led.history.push({ turn:turn, in:led.thisTurnIn, out:led.thisTurnOut, stockEnd:led.stock,
                         sources:_safeClone(led.sources), sinks:_safeClone(led.sinks) });
      if (led.history.length > GM.fiscal.meta.historyDepth) led.history.shift();
      led.lastTurnIn = led.thisTurnIn; led.lastTurnOut = led.thisTurnOut;
      led.thisTurnIn = 0; led.thisTurnOut = 0;
      led.sources = {}; led.sinks = {};
    });
  },
  annualArchive: function(year) {
    // 年度决算：汇总本年各回合 history → yearlyArchive[year]
  },
  _resolve: function(path) {
    var parts = path.split('.'); return GM.fiscal[parts[0]][parts[1]];
  }
};
```

**强制纪律**：所有改动库存的代码必须走 `Fiscal.*`，不得直接 `led.stock +=`。便于后续注入腐败/漂没/审计钩子。

### A.7 旧系统清理 / 迁移

- 新开游戏：不再写 `GM.vars.treasury` / `GM.vars.grain`
- 读老存档：`fullLoadGame` 中检测 `GM.vars.treasury/grain`，迁移为 `GM.fiscal.guoku.money/grain.stock`，原变量保留但打 `_legacy=true` 标记避免同步

- 诏令效果解析：原 `+treasury 100` 之类的硬编码效果作废，改为 `Fiscal.add('guoku.money', 100*10000, '赏赐')`

### C.5 征税公式（含自治度）

```js
amount = base × rate
       × (1 - corruption)
       × (1 - disasterLevel)
       × (1 - exemption)
       × (1 - disruption)
       × (1 - autonomy × retentionRate)
```

`retentionRate` 默认 0.8：完全自治区中央仅收到 20%（贡赋式象征性进账），但另外可通过 `tribute` 类税种获取定额藩属贡。

### C.6 每回合征收流程

```js
function Fiscal.collectTaxes(turn) {
  (sc.fiscalConfig.taxes || []).forEach(function(tax) {
    if (!tax.enabled) return;
    if (!_cadenceMatches(tax.cadence, turn)) return;
    _forEachTopLevelDivision(function(r) {
      var base = _getTaxBase(r, tax.baseType);
      var amount = base * tax.rate
                 * (1 - (r.corruption||0))
                 * (1 - (r.disasterLevel||0))
                 * (1 - (r.exemption||0))
                 * (1 - (r.disruption||0))
                 * (1 - (r.autonomy||0) * 0.8);
      if (amount <= 0) return;
      var path = tax.destination + '.' + tax.storeAs;
      Fiscal.add(path, amount, tax.name + '/' + r.name);
    });
  });
}
```

## 决策 D 详细方案（已定稿）

### D.1 核心原则

**玩家永远主控财政**：AI 只对玩家诏令被动响应，不主动产生 fiscalActions（选项 B）。

- 玩家明写金额 → AI 按数字扣
- 玩家写事由未写金额 → AI 估算
- 玩家未提 → AI **不得**自行加减账
- 固定扣（俸禄/军饷/日用）+ 税收 = 程序化，不走 AI

**制度改革判定分流**（跨系统统一，见 `设计方案-制度设计与演化.md` 第三节）：
- **AI 可直断**（诏书细节齐）→ 即时 fiscalActions
- **细节不足** → 触发二阶段（有司奏疏 / 求见）→ 方案定后才扣账
- 分流判定标准 = AI 能否从诏书提取足够实施信息，非"日常/制度"二分

### D.2 AI prompt 扩展（见设计文档 C.8 基础上 + 财政段落）

```
【当前财政现状】
帑廪：钱 5000000 贯，粮 3000000 石，布 120000 匹
内帑：钱 800000 贯，粮 50000 石
（换算：1 贯 = 1000 文；1 石 = 10 斗；1 匹 = 4 丈）
上回合进项：田赋 324000 石、丁税 82000 贯、...
上回合支出：俸禄 120000 贯、军饷 153000 贯、...

【本回合玩家诏令】<玩家输入>

【财政扣减判定任务】
仅针对玩家诏令中涉及钱粮布的条目判定：
- 明写数额：严格按数字
- 未明写但事由明确（赈灾/工程/赏赐/征伐）：按事项规模+库存+朝代惯例估算
- 诏令未涉及财政：不得产生 fiscalActions
- 单次扣款不得超过当前账本库存（硬顶）
- 库存不够：feasibility='insufficient'，只扣能扣的部分
- 大臣抵制强烈：feasibility='rejected'，一分不扣

返回 fiscalActions 数组，每条整数单位（1 贯/石/匹）：
{ account:'guoku'|'neicang', column:'money'|'grain'|'cloth',
  type:'spend'|'add'|'transfer', amount:<整数>,
  reason:'赈江南水灾', feasibility:'ok'|'insufficient'|'rejected',
  note:'户部告急，仅拨 5 万贯' }
```

### D.3 AI 返回 schema 扩展

主推演 JSON 新增 `fiscalActions` 数组（D.2 格式）。

### D.4 处理器 `_processFiscalActions`

```js
function _processFiscalActions(actions) {
  (actions||[]).forEach(function(a) {
    if (a.feasibility === 'rejected') {
      addEB('诏令驳回', a.reason + ' — ' + (a.note||'大臣抵制'));
      return;
    }
    var path = a.account + '.' + a.column;
    var stock = Fiscal.get(path);
    var safeAmount = a.amount;
    // 硬顶：单次 spend 不得超过当前库存
    if (a.type === 'spend' && safeAmount > stock) {
      safeAmount = Math.max(0, stock);
      a.feasibility = 'insufficient';
    }
    if (a.feasibility === 'insufficient') {
      addEB('库存告急', a.reason + ' — ' + (a.note||'部分执行'));
    }
    if (a.type === 'spend') Fiscal.spend(path, safeAmount, a.reason);
    else if (a.type === 'add') Fiscal.add(path, safeAmount, a.reason);
    else if (a.type === 'transfer') Fiscal.transfer(a.from, a.to, safeAmount, a.reason);
  });
}
```

### D.5 补给型 add 事件（抄家/贡赋/捐纳）

统一走 `fiscalActions` 数组（`type:'add'`），不独立字段。诏令未指定入帑廪还是内帑时，默认帑廪（抄家除外，可由诏令明写改为内帑）。

## 补充 P · 结构性规范

### P.1 cadence 与回合天数对应

| cadence 值 | 触发规则 |
|-----------|---------|
| `monthly` | 累计满 30 天自然天数触发一次（回合天数 ≤30 则计数；≥30 则每回合触发 1+次）|
| `quarterly` | 累计满 91 天触发（季末）|
| `biannual` | 累计满 182 天触发（春秋两季）|
| `annual` | 满 365 天（年末）|
| `event` | 事件驱动 |

实现：`Fiscal._cadenceMatches(tax, turn)` 维护 `tax._daysSinceLastTrigger`，每回合 += 剧本 daysPerTurn，满则触发并清零。

### P.2 多税叠加 + 层层加派（税负双层模型 · 王朝周期律内核）

#### 取消硬互斥
历史上一块田同时被多税征收（两税法夏秋两征）是常态。`group` 字段保留但改为"软提示"：编辑器提示"同组多选可能超配"，不硬禁止。

#### 三数字模型

每笔税有三个数字：

| 数字 | 含义 | 流向 |
|------|------|------|
| 名义应征 nominalTax | tax.rate × base 理想值 | — |
| 实际入库 stateCaptured | 扣漂没/腐败/自治/运损后到国库的值（原 C.6 公式）| `Fiscal.add` |
| **百姓实际负担 peasantActual** | 名义 × 各级加派倍数（新增）| `region.peasantBurdenStructure` |

#### 百姓实际负担公式

```js
peasantActual = nominalTax
  × (1 + countyMarkup)          // 县级加派（火耗/浮收/淋尖踢斛）
  × (1 + prefectureMarkup)       // 府级加派
  × (1 + circuitMarkup)          // 路/道/省级加派
  × (1 + landlordExploitation)   // 豪强盘剥（佃农被地主加收）
  × (1 + paymentConversionMult)  // 折纳价差（泛指：凡收取非所产迫使百姓换算的损耗——田赋折银/实物赋折钱/徭役折钱皆适用）
  × (1 + specialLevyMult)        // 中央特派（辽饷/剿饷/练饷）
```

#### 各级加派率决定因素

```js
countyMarkup = baseMarkup[level]
  × (1 - monitoringStrength[level])   // 都察院/御史/巡按越强越抑制
  × (1 + corruption[level])            // 本级腐败放大
  × (1 + poverty[level])               // 越穷越刮
  × (1 + centralLevyPressure)          // 中央加派 → 地方放大
```

#### region 数据结构

```js
region.peasantBurdenStructure = {
  nominal: 0,
  stateCaptured: 0,        // 国库实得
  peasantActual: 0,        // 百姓实际承受
  markupByLevel: {
    county: 0.15, prefecture: 0.08, circuit: 0.05,
    landlord: 0.30,
    paymentConversion: 0.20,   // 折纳价差（通用）—— 徭役/户口系统接入后由对应子系统动态覆写
    central: 0
  },
  pocketedByLocal: 0,        // 各级官吏截留
  pocketedByLandlord: 0,     // 豪强吞占
  pocketedByConverter: 0     // 粮商/官吏在折色过程中抽成
}
```

**留接口给未来的"徭役 / 户口"子系统**：
- `paymentConversion` 目前是静态配置或事件驱动
- 当 `在籍户口` 变量与徭役系统建立后，`paymentConversion` 的一部分（特指"徭役折钱"这一历史场景）将由徭役子系统动态计算——哪些朝代有徭役折钱、哪些没有、折钱率多少，全部由徭役系统决定
- 徭役子系统未建立前，此字段接受剧本静态配置或事件触发调整
- 接入点（将来）：`paymentConversion += corveeSys.getConversionRate(region, turn)`

#### UI 呈现

年度决算 · 各省分账新增"税负对照"栏：
```
江南  名义10% | 实际百姓负担 38% | 国库得 7.5% | 地方截流 19.5% | 豪强吞 11%
河北  名义10% | 实际百姓负担 27% | 国库得 5.8% | 地方截流  9.2% | 豪强吞 12%
朔方  名义10% | 实际百姓负担 18% | 国库得 1.8% | 自治留八成
```

#### 反馈联动

- 百姓实际负担 > 40% → 民心大降
- 某省 > 60% → 流民事件触发
- 全国均 > 50% → 农民起义风险拉满（王朝末期征兆）
- 玩家诏令"派御史"/"整饬吏治"/"清查火耗"→ 降 markupByLevel
- 中央加派（辽饷等）→ 层层放大，百姓负担非线性暴涨

## 二十一、与其他官方变量联动（2026-04-18 补完）

### 21.1 帑廪 ← 腐败（漏损层）

帑廪的"名义税额 vs 实征"差距来自**腐败系统**。

**接入点**：
```javascript
// 回合末计算实征率（取代原 D.1 中的静态估计）
function calcActualTaxRate(nominalTax) {
  var fc = GM.corruption.subDepts.fiscal.true;   // 税司腐败
  var pc = GM.corruption.subDepts.provincial.true; // 地方腐败
  var leakage = (fc + pc) / 200 * 0.7;  // 最高 70% 漏损
  return 1 - leakage;
}

// 举例：腐败 50 → 实征率 65%（35% 漏损）
//       腐败 80 → 实征率 44%（56% 漏损）
```

三数对照（民缴/官收/名义）中的差额直接由腐败决定。详见腐败系统 §3.1。

### 21.2 帑廪 ← 民心（顺从度）

民心决定百姓的**缴税顺从度**。

**接入点**：
```javascript
function calcTaxCollectionCompliance(region) {
  var m = GM.minxin.byRegion[region].true;
  return Math.max(0.3, m / 100 * 0.7 + 0.3);
  // m=80 → 1.0；m=50 → 0.85；m=30 → 0.65；m=10 → 0.35
}
```

实际实征 = 名义 × 腐败漏损 × 民心顺从度（两层折扣）。详见民心系统 §3.2。

### 21.3 帑廪 ← 皇权（调用权威）

皇权强弱直接影响帑廪的**可支配性**：
- 专制段（> 75）：税调灵活，但易征过度扰民
- 制衡段（35-75）：正常运作
- 权臣段（< 35）：地方截留严重，实际可用 ~ 50%

**接入点**：
```javascript
function calcFiscalDiscretion() {
  var h = GM.huangquan.index;
  if (h < 35) return 0.5;   // 权臣段：一半被截留
  if (h < 60) return 0.85;
  return 1.0;
}
```

### 21.4 帑廪 ← 皇威（暴君段虚账）

皇威暴君段的"过度执行"会让帑廪**账面增**但**实际虚**：
- 地方官争相报"超额完成税收指标"
- 但多出的部分来自浮收民间
- 民间实缴激增（民心崩盘）
- 未来反噬（民变→军费）

**接入点**：
```javascript
// 暴君段每季度执行
function applyTyrantFiscalDistortion() {
  if (GM.huangwei.index > 90) {
    GM.guoku.balance += GM.guoku.quarterlyIncome * 0.15;  // 账面 +15%
    GM.fiscal.floatingCollectionRate += 0.08;              // 浮收率激增
    GM.huangwei.tyrantSyndrome.hiddenDamage.fiscalBubble += GM.guoku.quarterlyIncome * 0.15;
  }
}
```

### 21.5 帑廪 → 民心（赋税反向）

原 P.2 已定义三数，此处补充民心反馈：
```javascript
// 民间实缴超过民间支付能力时 → 民心 -
function calcTaxMinxinImpact() {
  var actualPayment = GM.fiscal.peasantActualPayment;
  var capability = GM.minxin.payAbility;
  var ratio = actualPayment / capability;
  
  if (ratio > 1.0) return -Math.pow(ratio - 1, 1.5) * 15;
  return +3 * (1 - ratio);  // 轻税有正面影响
}
```
