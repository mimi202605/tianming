# 鼎革重构设计：民变实体化 · 玩家之死终局 · 继承机制 · CK3 化长期规划

> 2026-07-07 · owner 范式裁定驱动 · 四路勘察（民变链解剖/起义军实体/势力运行时生成/终局路径）支撑
> 状态：设计稿·待 owner 审定分波后开工

## 一、owner 裁定（铁律）

1. **民变=实体演进链**：民怨点火 → 地方出现**起义军**（真领袖、真军队）→ 攻城掠地成**新势力** → 可**建号立国** → 军事上**灭亡玩家政权**。绝非数值等级爬梯。
2. **终局=玩家角色被杀**这一具体事件，绝非任何数据阈值。
3. **被杀+有储君=继统续玩** → 继承机制升格为核心基石系统。
4. **禅代不死则游戏未终**（废帝/傀儡继续扮演）→ 为「扮演所有角色」（CK3 式）铺路。

## 二、地基勘察结论（2026-07-07 四路 opus·细节在勘察报告）

**病灶**：
- 三套民变系统互不相通零转化：`GM.minxin.revolts` 五级数值梯（7 条产线 schema 不统一·骰点爬梯·level5 撞线 `_gameOver`）、`GM._activeRevolts`（AI 叙事驱动的实体模型·有真领袖角色/organizationType/territoryControl/transform 转化机·但军力是纸面数字）、`GM.rovingRebels`（确定性流寇·纯强度数字+省名·灭不了国）。
- 「玩家角色被杀」无统一表达：**两条静默杀漏洞**（御驾亲征战殁/被俘只标 `alive=false` 不继承不终局；AI 判弑君成功只记史册人照活）；三处阈值终局（民变5级/篡位骰点/dynastyReplaced 叙事信号）；终局 UI 分裂两套（`_playerDead` 简版屏 vs `_showEndgameScreen` 富屏）。
- `resolveHeir` 不看死因（被弑与老死同路），fallback「同势力最强者」可让弑君者本人继统。

**可复用的底盘（全部就位）**：
- 军事：`GM.armies` 敌军先例成熟（后金军=一等军队条目·faction 归属字段）、确定性战斗结算（BattleEngine.resolve）、非玩家 vs 非玩家能打（battle-turn 按 faction 分侧）、攻占易主 API（applyBattleResult→setProvinceOwner→FactionMembership.assignProvince 三源同步）、御驾亲征只要求「faction≠玩家的真军队」。
- 势力：运行时新建 faction 两条生产路径已在（`faction_create`@apply:2443 / `revolt_transform:toFaction`@apply:2302）；读侧消费者绝大多数动态遍历 GM.facs 自动接纳；关系矩阵懒建；新势力自动入档、agent/prompt/derived 自动接纳。
- 继承：resolveHeir（册立太子 designatedHeirId 最高优先/默认嫡长/兄终弟及/选贤）+ 皇嗣国本（诞育入宗牒）已有。

**已定位的裂缝**：
- 领土双表示：name-keyed（assignProvince 一处写四源·全通）vs id-keyed（adminHierarchy 按 fac.id 分桶·**运行时无开桶 API**）——现有两条建政路径两套都不写=「无地政权」。
- toFaction 路径漏 `addToIndex`；borderRisk 对 NPC 势力 facId 当 name 用；salience cap24 可能截断弱小新势力。
- adminHierarchy GM./P. 双引用+快照持久化——运行时新桶要挂对对象。

## 三、目标架构

**叛军单一实体模型**（升级 `_activeRevolts` 为主模型）：
```
民怨点火(确定性/事件/AI 三源) → RevoltFactory 生成实体：
  领袖 char(已有) + 起义军 GM.armies 条目(新) + revolt 实体(已有 schema)
→ 演进(确定性阶梯+AI 叙事双驱): flowingBandit→baseArea→builtState
  · 攻占府州=真易主(assignProvince+adminHierarchy)
  · 剿抚=真军事结算(官军 vs 起义军·可御驾亲征)
→ builtState 建政 = 运行时新 faction(领土两套齐写)
→ 兵锋指京师 → 破京事件 → 天子下场(被杀/被俘/出逃) → 玩家之死裁决器
```

**玩家之死裁决器**（`adjudicatePlayerDeath(ch, cause, opts)`·落 tm-endturn-helpers）：
```
一切死亡产地(AI 叙事杀/自然老死/战殁/被弑/破京被戮)只调此一口
→ resolveHeir 有合法继统 → 世代传承(继统事件·换角色续玩)
→ 绝嗣/无合法继统 → 终局(富屏·太史公·真亡因)
被俘/被废/傀儡 ≠ 死 → 特殊状态继续扮演(不终局)
```
三处阈值 `_gameOver` 全部退役：民变5级(随数值梯退役)、篡位骰点(反转为废立事件·玩家活着)、dynastyReplaced(转译为破京军事链)。

## 四·五、帝王本纪系统（owner 2026-07-07 提案·与 R1e 终局富屏合流）

> owner：「玩家死或亡国都走太史公，让 AI 详细看他的整局活动和时政记实录等内容，一部分一部分生成玩家的本纪，严格按本纪、实录文言文撰写。」**时序裁定（2026-07-07 订正）**：不是游戏进行中定期生成——是**玩家死时（终局时刻）回顾式修史**：在位多少回合，按**每 12 回合一段**切块逐段发起 AI 调用（玩 36 回合驾崩=3 次调用成三卷；6 回合太短已弃）。

**定性**：史官层级的收官件——起居注(qijuHistory·已有)→时政记/实录(shizhengji/shilu·已有)→**本纪(终局回顾式修纂·新)**→太史公论赞(已有·升级)。正合官修史真实流程：**实录本就是皇帝死后按起居注修纂的**——终局时刻修本纪比生前逐段更合体例。12 回合≈一年，**天然一年一卷、按年编次**。

**设计**：
- **触发**：玩家之死裁决器/亡国终局（R1e 富屏）→ `TM.Benji.compose(GM)` 修纂 orchestrator。**无 per-turn tick**。
- **分段生成**：在位 N 回合 → ceil(N/12) 段，**串行**逐段调用（段间衔接：前段末 ~200 字入下段 prompt）；末段并入死亡/亡国之事+「赞曰」论赞。终局屏**逐卷上屏**（「史官修纂中……第 X 卷成」的进度体验·先显既有太史公短评与指标不空等）。
- **输入组装**（每段 12 回合窗）：该窗的起居注条目+史记 shizhengji/shilu 摘要+编年 biannianItems+玩家决策（诏令/批红）+大事（战和/灾异/人事/民变）+roster 防幻觉名单。
- **★史料保全前提（实施第一件事）**：终局才修史⇒全程史料必须留到死——须核 qijuHistory/shijiHistory/biannianItems 等子树的滚动 cap 与保留策略，早期回合素材不可被裁掉；与「存档瘦身三案」联动权衡（如剥预渲染 html 换保结构化文本）。素材缺失的段落用编年骨架兜底并如实短卷。
- **体例 prompt**：仿《实录》/《史记》本纪——编年直书（「X年春正月」纪月）、称「上」、据实不虚构、不加评论；「赞曰」唯末卷。有「史记三段成文」sc2Pipeline 先例可循。输出校验：文言/长度带/人名对账（不识之名重试）。
- **终局合流（接 R1e）**：富屏=本纪全卷+太史公论赞——太史公评语的输入从「failGoal.title 一句话」升级为**读完整本纪成文**（勘察 D 指出的评语贫瘠问题根治）。
- **成品留存**：写 `GM._benji`（终局档随存档）+ 存战绩侧（接 A1 tm_playHistory 战绩留痕）——主菜单「历代亲历」可重读历局帝王本纪，天然的收藏/留存功能。
- **成本**：终局一次性 ceil(N/12) 次中等调用（输入摘要化 ~3-5K·输出 ~800 字/卷）串行·长局（120 回合=10 卷）需数分钟——逐卷上屏消化等待。flag `benjiEnabled` 默认关+设置开关；无 key 诚实不生成。

**实施位置**：随 Wave R1 之后单独一刀（R1g·依赖 R1e 富屏合一），不阻塞 R2。

## 六、长期规划：CK3 式「扮演所有角色」

> owner 方向：像 CK3 那样允许玩家扮演所有角色。扮演的是「人」，不是「位」。

**阶段 0（Wave R1 即达成）——角色≠皇位**：终局只系于角色之死与绝嗣；被废、被俘、禅代都是人生章节。继承=换一个「人」继续玩。

**阶段 1——失位可玩态（废帝/傀儡/北狩）**：身份状态机（在位/傀儡/废帝/被俘/太上皇）。核心架构课题：**五渠道是皇帝的操作面**——傀儡=渠道被劫持（批红被代批·诏令出不了宫门）、废帝=只剩问对/密信/密谋（复辟线玩法）、被俘=囚中书信。操作面第一次「按身份装配」，是渠道化改造的试验田。

**阶段 2——视角切换基建**：世代传承已做 isPlayer 转移，推广为通用 `possessCharacter(ch)`（玩家身份=charId 单一真源·P.playerInfo 派生）。AI 叙事视角化：sysP 以「玩家所扮之人」为中心重构（现在整个 prompt 体系以皇帝为中心——这是 CK3 化最大的 AI 侧工程）。

**阶段 3——非帝开局身份**：权臣线（挟天子·代批·党羽·终点=禅代开新朝或还政）、藩王线（就藩·靖难）、起义领袖线（**R2 的起义军实体就是这条线的全部地基**——从另一侧扮演李自成打天下）。每身份=渠道集+目标轴+终局定义（都系于角色之死）。
- 数据基建大多已有：官制树/党派/藩王(宗藩)/起义实体/军事。缺的是「非皇帝操作渠道」与「AI 以非帝为主角推演」。

**阶段 4——全角色扮演**：任意 char 可玩·人生事件流·家族传承（clan 共财/恩荫/婚姻已有底子）·换代继承跨身份（子承父业的知府之家）。

**排序建议**：R1→R2 是必经之路且本身就是阶段 0；阶段 1 借 R1d 废帝态顺势起步；阶段 2 的 possess 基建做完，阶段 3 的三条身份线可按剧本价值逐条上（起义领袖线最便宜——地基全在 R2 里）。
